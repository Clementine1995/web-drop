## Babel 是如何编译 Class 的

ES6的 Class 可以看作一个语法糖，绝大部分功能ES5都可以实现。

+ 关于 constructor，ES5的构造函数相当于类的 constructor 方法。不过：类的内部所有定义的方法，都是不可枚举的
+ 实例属性，ES5中是写在构造函数内，并利用this，而ES6类中类似只不过是写在 constructor 内的。当然新的提案中也可以简写

  ```js
    class Person {
      state = {
        count: 0
      };
    }
  ```

+ 静态方法，ES6中通过static修饰的方法就是静态方法，而ES5中就是通过给构造函数添加属性的方式实现
+ 静态属性，ES6中通过static声明的属性就是静态属性，ES5中通过给构造函数添加属性实现
+ new调用，class是必须通过new调用的，而构造函数不必
+ getter 和 setter，ES5在原型中声明，而ES6在类声明中。

### 编译关于必须使用new调用

```js
// _classCallCheck 的作用是检查 Person 是否是通过 new 的方式调用
function _classCallCheck(instance, Constructor) {
  if (!(instance instanceof Constructor)) {
    throw new TypeError("Cannot call a class as a function");
  }
}
var Person = function Person(name) {
    _classCallCheck(this, Person);
    this.name = name;
};
```

### 编译实例与静态属性

这个跟上面说的一样，写在 constructor 与写在外面的实例属性，都会通过this绑定到实例上，而静态属性是绑定到了构造函数上

```js
var Person = function Person(name) {
  _classCallCheck(this, Person);

  this.foo = 'foo';

  this.name = name;
};
Person.bar = 'bar';
```

### 编译静态方法实例方法以及getter/setter

Babel 生成了一个 _createClass 辅助函数，该函数传入三个参数，第一个是构造函数，在这个例子中也就是 Person，第二个是要添加到原型上的函数数组，第三个是要添加到构造函数本身的函数数组，也就是所有添加 static 关键字的函数。_createClass 中有生成了 defineProperties 辅助函数，使用 Object.defineProperty 方法添加属性。

```js
var _createClass = function() {
  function defineProperties(target, props) {
    for (var i = 0; i < props.length; i++) {
      var descriptor = props[i];
      descriptor.enumerable = descriptor.enumerable || false;
      descriptor.configurable = true;
      if ("value" in descriptor) descriptor.writable = true;
      Object.defineProperty(target, descriptor.key, descriptor);
    }
  }
  return function(Constructor, protoProps, staticProps) {
    if (protoProps) defineProperties(Constructor.prototype, protoProps);
    if (staticProps) defineProperties(Constructor, staticProps);
    return Constructor;
  };
}();
```

### ES6的extend

ES5中实现继承方法有很多，寄生组合式继承是其中比较好的一种办法。而相比于ES5的寄生组合式继承，ES6的extend关键字，更清晰直观

```js
class Parent {
  constructor(name) {
    this.name = name;
  }
}
class Child extends Parent {
  constructor(name, age) {
    // super 关键字表示父类的构造函数，相当于 ES5 的 Parent.call(this)。
    // 子类必须在 constructor 方法中调用 super 方法，否则新建实例时会报错。
    // 这是因为子类没有自己的 this 对象，而是继承父类的 this 对象，然后对其进行加工。
    // 如果不调用 super 方法，子类就得不到 this 对象。
    super(name); // 调用父类的 constructor(name)
    this.age = age;
  }
}
var child1 = new Child('kevin', '18');
console.log(child1);
```

### 子类的 __proto__

ES6中父类的静态方法是可以被子类继承的，因为 Class 作为构造函数的语法糖，同时有 prototype 属性和 __proto__ 属性，因此同时存在两条继承链。相比寄生组合式继承，ES6 的 class 多了一个 Object.setPrototypeOf(Child, Parent) 的步骤。

1. 子类的 __proto__ 属性，表示构造函数的继承，总是指向父类。
2. 子类 prototype 属性的 __proto__ 属性，表示方法的继承，总是指向父类的 prototype 属性。

注意：Object.setPrototypeOf(Child, Parent)是为了继承父类的静态属性，Child instanceof Parent 并不为真

### 继承目标

extends 关键字后面可以跟多种类型的值。下面例子中的A，只要是一个有 prototype 属性的函数，就能被 B 继承。由于函数都有 prototype 属性（除了 Function.prototype 函数），因此 A 可以是任意函数。

```js
class B extends A {
}
```

除了函数之外，A 的值还可以是 null，当 extend null 的时候：

```js
class A extends null {
}
console.log(A.__proto__ === Function.prototype); // true
console.log(A.prototype.__proto__ === undefined); // true
```

关键代码：

```js
function _inherits(subClass, superClass) {
  // extend 的继承目标必须是函数或者是 null
  if (typeof superClass !== "function" && superClass !== null) {
    throw new TypeError("Super expression must either be null or a function, not " + typeof superClass);
  }
  // 类似于 ES5 的寄生组合式继承，使用 Object.create，设置子类 prototype 属性的 __proto__ 属性指向父类的 prototype 属性
  subClass.prototype = Object.create(superClass && superClass.prototype, { constructor: { value: subClass, enumerable: false, writable: true, configurable: true } });
  // 设置子类的 __proto__ 属性指向父类
  if (superClass) Object.setPrototypeOf ? Object.setPrototypeOf(subClass, superClass) : subClass.__proto__ = superClass;
}

function _possibleConstructorReturn(self, call) {
  if (!self) {
    throw new ReferenceError("this hasn't been initialised - super() hasn't been called");
  }
  // 对于 Parent.call(this) 的值，如果是 object 类型或者是 function 类型，就返回 Parent.call(this)
  // 如果是 null 或者基本类型的值或者是 undefined，都会返回 self 也就是子类的 this。
  return call && (typeof call === "object" || typeof call === "function") ? call : self;
}

var Child = function(_Parent) {
  _inherits(Child, _Parent);

  function Child(name, age) {
    _classCallCheck(this, Child);

    // 调用父类的 constructor(name)
    var _this = _possibleConstructorReturn(this, (Child.__proto__ || Object.getPrototypeOf(Child)).call(this, name));

    _this.age = age;
    return _this;
  }
  return Child;
}(Parent);
```

具体步骤：

1. 首先执行 _inherits(Child, Parent)，建立 Child 和 Parent 的原型链关系，即 Object.setPrototypeOf(Child.prototype, Parent.prototype) 和 Object.setPrototypeOf(Child, Parent)。
2. 然后调用 Parent.call(this, name)，根据 Parent 构造函数的返回值类型确定子类构造函数 this 的初始值 _this。
3. 最终，根据子类构造函数，修改 _this 的值，然后返回该值。

## defineProperty 与 proxy

defineProperty用来详细的配置一个对象的某个属性的表现，比如是否可配置（configurable），是否可枚举（enumerable），还有数据描述符以及存储描述符（get,set），其中数据描述符还可以配置是否可写。而其中通过**存储描述符**拦截对象属性存取就是监听数据变化的关键。下面是一个简单的watch函数。

```js
(function(){
  var root = this;
  function watch(obj, name, func){
    var value = obj[name];

    Object.defineProperty(obj, name, {
      get: function() {
        return value;
      },
      set: function(newValue) {
        value = newValue;
        func(value)
      }
    });
    if (value) obj[name] = value
  }
  this.watch = watch;
})()
```

而 Proxy 可以重定义更多的行为，具体可以看深入理解ES6，额这本书。下面是proxy优化过的watch

```js
(function() {
  var root = this;
  function watch(target, func) {
    var proxy = new Proxy(target, {
        get: function(target, prop) {
            return target[prop];
        },
        set: function(target, prop, value) {
            target[prop] = value;
            func(prop, value);
        }
    });
     return proxy;
  }
  this.watch = watch;
})()
```

defineProperty 与 proxy 这两种方法总体来看是 Proxy 更好，Vue3中就使用 Proxy 代替了defineProperty，不过 Proxy 存在兼容性问题。

+ Proxy代理的是对象，defineProperty代理的是属性
+ Proxy可以拦截数组变化，而defineProperty无能为力
+ Proxy拦截方法很多，而defineProperty只是通过get/set
+ Proxy返回一个拦截后的数据，defineProperty直接修改原始数据

## 模块加载方案

随着模块化的发展，模块加载方案主要有 AMD，CMD，CommonJS，ES6 模块。

AMD与CMD都是模块定义的一种规范，分别是 requirejs 与 seajs 在推广过程中对模块定义的规范化产出。只要按照规范编写模块与依赖，就能正确解析并执行。它们的区别如下：

1.CMD 推崇依赖就近，延迟执行，AMD 推崇依赖前置，提前执行。看两个项目中的 main.js：

```js
// require.js 例子中的 main.js
// 依赖必须一开始就写好
require(['./add', './square'], function(addModule, squareModule) {
  console.log(addModule.add(1, 1))
  console.log(squareModule.square(3))
});
// sea.js 例子中的 main.js
define(function(require, exports, module) {
  var addModule = require('./add');
  console.log(addModule.add(1, 1))
  // 依赖可以就近书写
  var squareModule = require('./square');
  console.log(squareModule.square(3))
});
```

2.对于依赖的模块，AMD 是提前执行，CMD 是延迟执行。看两个项目中的打印顺序：

```console
// require.js
加载了 add 模块
加载了 multiply 模块
加载了 square 模块
2
9
// sea.js
加载了 add 模块
2
加载了 square 模块
加载了 multiply 模块
9
AMD 是将需要使用的模块先加载完再执行代码，而 CMD 是在 require 的时候才去加载模块文件，加载完再接着执行。
```

3.同时它们的导出方式也不同

### CommonJS

服务器端比如 node，采用的则是 CommonJS 规范。它与 seajs 相似，都是在 require 的时候才去加载模块文件，加载完再接着执行。只不过导入导出方式不同

```js
// 导出模块的方式：
var add = function(x, y)
    return x + y;
};
module.exports.add = add;

// 引入模块的方式：
var add = require('./add.js');
console.log(add.add(1, 1));
```

注意：CommonJS 规范加载模块是同步的，也就是说，只有加载完成，才能执行后面的操作。而AMD规范是非同步，可以指定回调函数。Nodejs 主要用于服务器编程，模块也一般在本地，不太需要考虑非同步加载，所以CommonJs规范比较适用，但是浏览器环境要从服务端加载模块，就要采用非同步的AMD规范。

### ES6模块

ECMAScript2015 规定了新的模块加载方案。导入导出也与别的不同。

```js
var firstName = 'Michael';
var lastName = 'Jackson';
var year = 1958;
export {firstName, lastName, year};
// 还有 epxort default 的形式

import {firstName, lastName, year} from './profile';
```

注意：浏览器加载 ES6 模块，也使用 `<script>` 标签，但是要加入 type="module" 属性。它的加载效果与 requirejs 相同，也就是将需要使用的模块先加载完再执行代码。

>补充阅读[ES modules: A cartoon deep-dive](https://hacks.mozilla.org/2018/03/es-modules-a-cartoon-deep-dive/)

#### ES6 模块是如何工作的

当在使用模块时，需要建立一张依赖图，不同的依赖之间就是靠着 import 语句连接起来的，通过这些 import 语句浏览器或者 Node 可以清楚的知道需要加载的代码，而只需要提供一个入口文件，之后就会根据那些 import 声明找到剩余的代码。

这些文件本身是不能被浏览器使用的，它们需要被转换成**模块记录**这一种数据结构，在之后模块记录会被转换成一个**模块实例**，一个模块实例由代码与状态构成。代码也就是一系列的指令，而状态是变量在任何时间点的实际值，当然这些变量只是作为内存中存储这些值的别名而已（引用）。

每一个模块都需要有一个模块实例，模块加载的过程也就是从入口文件开始，构建整个模块实例关系表，对于每个模块分为下面三步：

1. 构建：查找、下载以及将所有文件解析成模块记录。
2. 实例化：给所有导出的值在内存中分配空间（并不赋值），然后把导入与导出的点与这些内存关联起来。这被称作链接。
3. 求值：运行代码，把内存中的空间赋予变量真实的值。

可以把 ES模块看作是异步的，因为整个过程被分为加载，实例化，求值三个阶段，并且这三个阶段也是可以分开执行的，这与 CommonJS 是不同的，在 CommonJS 中，一个模块的依赖是在模块加载之后才立刻进行加载、实例化、求值的，中间不会有任何的打断，也就是同步。

ES 模块描述了应该如何将这些文件解析成模块记录，以及应该如何去实例化和进行求值。但是，它没有说明如何去首先获得这些文件。不同的平台有不同的加载器，加载器精确控制模块的加载方式，它称为 ES模块方法，ParseModule,Module.Instantiate,以及Module.Evaluate，下面来看每个步骤中的具体细节

##### 构建过程

构建阶段每一个模块发生了三件事：

1. 判断应该从何处下载文件所包含的模块（又叫模块解决方案）。
2. 获取文件（通过 url 下载 或者 通过文件系统加载）
3. 将文件解析进模块记录

+ 查找到文件然后获取到它

加载器将会尽可能的去找到文件然后去下载它。首先要去找到入口文件。在 HTML 中，应该通过 script 标签告诉加载器入口文件在哪，使用 type="module"来指明一个js 文件是一个模块。然后通过 import 语句中的**模块声明**找到下一个引用的模块，如果有的话。

关于模块声明，在浏览器端和 Node 端有不同的处理方式。每一个宿主环境有它自己的方法去解释用来模块声明的字符串。为了完成这个，模块声明使用了一种叫做模块解释的算法去区分不同的宿主环境。目前浏览器只能接受 URLs 作为模块声明。对于整个模块图来说，加载模块不是一个同步行为。无法知道哪一个依赖需要去获取直到把整个文件都解析完成，以及只有等获取到文件才能开始解析它。

这意味着必须逐层遍历该树，解析一个文件，然后找出其依赖项，然后查找并加载这些依赖项。如果主线程在等待每一个文件下载，那么其他的任务将会排在主线程事件队列的后面。这就是为什么在浏览器中下载这些部分会花费很长时间

这样阻塞主线程会使使用模块的应用程序使用起来太慢。这也就是ES模块规范将算法分为多个阶段的原因之一。把构造阶段拆分出去，使浏览器可以在开始实例化的同步工作之前获取文件并增强对模块图的理解。这种拆分成多阶段也是 ES模块与 CommonJS 关键不同点。

CommonJS 与 ES modules 的不同是，通过文件系统去加载文件，这比从网上下载文件要花的时间少得多。这就意味着，Node 将会在加载文件的时候阻塞主线程。只要文件加载完成，它就会去实例化并且去做求值操作（这也就是 CommonJS 不会在各个独立阶段去做的原因）。这同样说明了，当在返回模块实例之前，就会遍历整个依赖关系树然后去完成加载、实例化以及对各个依赖进行求值的操作。

注意，在 Node 中使用 CommonJS 模块时，在模块声明中是可以使用变量的，因为在查找下一个模块之前，将执行完这个模块所有的代码（直到通过require去返回这个声明）。这就意味着这些变量将会在处理模块解析时被赋值。

但是在 ES modules 中，将在执行模块解析和进行求值操作前就建立好整个模块依赖关系图表。这也就是说在模块声明时，不能去使用这些变量，因为这些**变量那时还并没有被赋值**，并且在 import 之前的语句会在 import 之后才执行。

但是有的时候有非常需要去使用变量作为模块声明，举个例子，可能会存在的一种情况是需要根据代码的执行效果来决定你需要引入哪个模块。为了能在 ES modules 这么去做，于是就存在一种叫做动态引入的提议，就像这样：

```js
import(`${path}/foo.js`)
```

这种通过import()去加载任意文件的方法是把它作为每一个单独的依赖图表的入口。这种动态引入模块会开始一个新的被单独处理的图。

有一点要注意的是，对于任意模块而言所有的这些图都共享同一个模块实例。这是因为加载器会缓存这些模块实例。对于每一个模块而言都存在于一个特殊的作用域内，这里面仅仅只会存在一个模块实例。显然，这会减少引擎的工作量。举个例子，目标模块文件只会被加载一次即使此时有多个模块文件都依赖于它。loader 通过**模块映射集合**来管理这个缓存。每个全局变量在单独的模块映射集合中跟踪其模块（集合大体结构，key为url，value为状态，比如 fetching，如果是已经获取到了，那就是模块记录）。当加载器准备去获取一个 URL 的时候，它会将这个 URL 放入模块映射中，然后对当前正在获取的文件做一个标记。然后它将发送一个请求（状态为 fetching），紧接着开始准备开始获取下一个文件。当其他模块也依赖这个同样的文件时，加载器将会在模块映射集合中去遍历这个 URL，如果它发现这个文件正在被获取，那么加载器会直接查找下一个 URL。但是模块映射集合并不会去保存已经被获取过的文件的栈。

+ 解析过程

现在已经获取到了这个文件，需要将它解析为一条模块记录。这会帮助浏览器知道这些模块不一样的部分。创建了模块记录之后，它会被放入**模块映射集合**中，后面无论何时再次请求它，就可以直接从这个集合中取。需要注意的是，所有的模块被解析后都会被当做在顶部有`use strict`。还有另外两个细节，await关键词会被预先储备到模块代码的最顶部，以及顶级作用域中this是undefined。

这种不同的解析方式称为“解析目标”。如果解析相同的文件但使用了不同的目标，那么最终将得到不同的结果。 因此想在开始解析之前先知道要解析的文件类型，无论它是不是一个模块。在浏览器中，这将非常的简单，只需要在 script 标签中设置type="module"。在 Node 端，不会使用到 HTML 标签，所以没办法去使用type属性。社区为此想出了一个解决办法，对于这类文件使用了mjs的扩展名。通过这个扩展名告诉 Node，“这是一个模块”。无论哪种方式，加载器将会决定是否将一个文件当做模块去处理。如果这是一个模块并且存在 import 语句，则它将重新开始该过程，直到获取并解析了所有文件。

##### 实例化

实例化将代码与状态结合在一起。而状态存在于内存中，因此实例化步骤就是将所有内容连接到内存。

首先JS引擎创建一个模块环境记录，它会为模块记录管理变量，然后它会为所有的 export 语句在内存中开辟空间，模块环境记录将会跟踪内存中与每个 export 相关联的区域。直到进行求值操作的时候这些内存区域才会被填充真实的值。有个例外：在此阶段中所有导出的函数声明将会被初始化。

为了实例化模块图，引擎将进行**深度优先遍历**。这意味着它将下降到图表的底部-直到不依赖其他项的依赖项-并设置其导出。

引擎完成了这个模块下所有导出的串联——模块依赖的所有导出。要注意的是导出和导入指向内存中同一块区域。将所有导出都串联起来保证了所有的导入能和与它对应的导出匹配。

这不同于 CommonJS 的模块化。在 CommonJS 中整个导出的对象是导出的一个复制。这就意味着，所有的值（比方说数字）都是导出值的复制。这意味着，如果导出模块以后更改了该值，则导入模块将看不到该更改。

相反，ES模块使用一种称为实时绑定的东西。两个模块都指向内存中的相同位置。这意味着，当导出模块更改值时，该更改将显示在导入模块中。导出值的模块可以随时修改导出的值，而导入模块的不能改变它们导入的值（导入的值会变成 const 的），但是如果导入的是一个对象，那对象属性是可以更改的。之所以拥有这样的实时绑定，是因为可以在不运行任何代码的情况下连接所有模块。当具有循环依赖性时，这将有助于求值。因此，在此步骤结束时，已连接了所有实例以及导出/导入变量的存储位置。

##### 求值操作

最后一步是对内存中的相关区域进行填充。JS 引擎是通过执行顶层代码去完成这件事的——在函数外的代码。

除了对内存中相关进行填充外，对代码进行求值也会造成副作用。比如说，模块可能会去调用一个服务。因为潜在的副作用，所以想要只对模块进行一次求值。通过使用**模块映射集合**缓存每个URL，因此每个模块只有一个模块记录，这就保证了每一个模块只被执行一次。和实例化的过程一样，它同样采用的是**深度优先**后序遍历的方法。

同时这种实时绑定也支持了循环依赖。

### ES6模块与 CommonJs

它们有两个**重大差异**

1. CommonJS 模块输出的是一个**值的拷贝**，ES6 模块输出的是值的引用。CommonJS 模块输出的是值的拷贝，也就是说，一旦输出一个值，模块内部的变化就影响不到这个值。ES6 模块是动态引用，并且不会缓存值，模块里面的变量绑定其所在的模块。
2. CommonJS 模块是运行时加载，ES6 模块是编译时输出接口。因为 CommonJS 加载的是一个对象（即module.exports属性），该对象只有在脚本运行完才会生成。而 ES6 模块不是对象，它的对外接口只是一种静态定义，在代码静态解析阶段就会生成。

CommonJS 情况下，counter仅仅是一个原始值类型，上面也说到CommonJS 模块输出的是一个**值的拷贝**，所以输出后，内部模块变化影响不到输出的值，当然这里如果换成引用类型，比如把 counter 包在一个对象里导出就会影响到。

```js
// 输出模块 counter.js
var counter = 3;
function incCounter() {
  counter++;
}
module.exports = {
    counter: counter,
    incCounter: incCounter,
};
// 引入模块 main.js
var mod = require('./counter');

console.log(mod.counter);  // 3
mod.incCounter();
console.log(mod.counter); // 3
```

ES6模块情况如下：

```js
// counter.js
export let counter = 3;
export function incCounter() {
  counter++;
}

// main.js
import { counter, incCounter } from './counter';
console.log(counter); // 3
incCounter();
console.log(counter); // 4
```

ES6 模块的运行机制与 CommonJS 不一样。JS 引擎对脚本静态分析的时候，遇到模块加载命令 import，就会生成一个只读引用。等到脚本真正执行时，再根据这个只读引用，到被加载的那个模块里面去取值。换句话说，ES6 的 import 有点像 Unix 系统的“符号连接”，原始值变了，import 加载的值也会跟着变。因此，ES6 模块是动态引用，并且不会缓存值，模块里面的变量绑定其所在的模块。

### 补充有关package.json中的导出

#### nodejs中加载模块

+ .mjs文件总是以 ES6 模块加载
+ .cjs文件总是以 CommonJS 模块加载
+ .js文件的加载取决于package.json里面type字段的设置

#### main 字段

package.json 文件中两个字段可以指定模块的入口文件：main和exports

```json
//  如果没有type字段，index.js就会被解释为CommonJS模块
{
  "type": "module",
  "main": "./src/index.js"
}
```

#### exports 字段

exports 字段的优先级高于main字段，但是该字段在一些打包工具中并没有得到支持，比如webpack4就不支持，webpack 5 beta 支持该字段

```json
// ./node_modules/es-module-package/package.json
{
  "exports": {
    "./submodule":  "./src/submodule.js"
  }
}
// 上面代码指定src/submodule.js别名为submodule,然后就可以从别名加载这个文件

import submodule from 'es-module-package/submodule'
// 加载 ./node_modules/es-module-package/src/submodule.js
```

### Babel与Webpack对模块的编译与打包

首先Babel 只是把 ES6 模块语法转为 CommonJS 模块语法，然而浏览器是不支持这种模块语法的，没有 module、 exports、 require 等环境变量。这个时候就需要Webpack，它会将编译后的代码再包一层，注入这些不存在的变量，使浏览器可以运行。

下面是有关CommonJs的小例子以及思考

```js
/*************** child.js**********************/
let foo = 1
setTimeout(()=>{
  foo=2;
  exports.foo= foo
},1000)
exports.foo=foo

/*******************index.js***************************/
var test =require('./child');
console.log(test.foo);// 1
setTimeout(()=>{
  console.log(test.foo) // 2
},2000)
```

在node.js中模块导出内容时 exports 就是 module.exports 的引用。 node.js 一个模块引入另外一个模块的变量的时候就是获取的 module.exports 上导出的内容。
上面这个例子因为最初导出的内容是 { foo: 1 } ，然后一秒后将其改成了 { foo: 2 }，它与之前导出的对象联系还在，用的还是同一个引用，而下面这个例子，一秒后将exports 赋值为一个新的对象，更换了引用，这时最初导出去的对象已经与一秒后的对象没有任何联系，所以也不会发生变化。

```js
/*************** child.js**********************/
let foo = 1

setTimeout(()=>{
  foo=2;
  module.exports={foo}
},1000)
module.exports={foo}

/*******************index.js***************************/
var test =require('./child');

console.log(test.foo);// 1

setTimeout(()=>{
  console.log(test.foo) // 1
},2000)
```

## 私有变量的实现

>具体查看[私有变量的实现](https://github.com/mqyqingfeng/Blog/issues/110)

+ 约定_xx表示私有变量，这种办法仅仅是约定，并且易于理解，但是外部可以随意修改，也可以被遍历到
+ 利用闭包，可以通过在 constructor 中声明变量，然后提供方法可以访问该变量，解决了外部访问的问题，但是导致 constructor 变得复杂，方法也只存在于实例上，无法被 super 调用。还可以通过立即执行方式来定义私有变量，不必写在 constructor 中，比直接写在 constructor 更轻便一点。不过立即执行的方法，会导致公用一个私有变量，后面新创建的实例传入的初始化值会覆盖之前实例的对应属性。这一点不合理。
+ Symbol，还是立即执行的方式，只不过私有变量key用Symbol生成。
+ WeakMap，可以利用 WeakMap key为对象的特性，保存当前实例与其上的值。
+ 最新提案，#修饰符，用于表示私有属性。
